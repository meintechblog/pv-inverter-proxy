## Interaction Contract

### Banner Visibility Logic

| Condition | Banner State |
|-----------|-------------|
| `venus_os_detected == false` | Hidden (`display:none`) |
| `venus_os_detected == true` AND `venus-host` input is empty | Visible (`display:block`) |
| `venus_os_detected == true` AND `venus-host` input has value | Hidden (`display:none`) |

**Trigger:** WebSocket snapshot includes `venus_os_detected` field. The `handleSnapshot` function calls `updateAutoDetectBanner(snapshot)` on every cycle (1s interval).

**Entrance animation:** Use existing `ve-slide-up` keyframe animation via `ve-card--entering` class applied once when banner transitions from hidden to visible. Remove the class after animation completes (300ms).

**No exit animation:** Banner hides instantly via `display:none` when condition no longer met.

### User Flow

1. Venus OS sends Modbus write to Model 123 registers
2. Backend sets `shared_ctx["venus_os_detected"] = True`
3. Next WebSocket snapshot includes `venus_os_detected: true`
4. Frontend shows banner above config form (if venus-host is empty)
5. User scrolls down (or is already on config page) and sees banner
6. User enters Venus OS IP in the existing `#venus-host` input field
7. Banner hides immediately (input now has value)
8. User clicks "Save & Apply" (existing button)
9. Config saves, MQTT connection starts, connection bobble turns green

### No New Buttons or Forms

The banner is purely informational. It directs the user to the existing Venus OS IP input field that already exists in the config form. No "Test & Apply" button is added to the banner itself — the existing "Save & Apply" button at the bottom of the config form serves this purpose.
